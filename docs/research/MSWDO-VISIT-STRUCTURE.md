# MSWDO Tagudin — Compact Visit Structure

**Purpose:** Researcher guide for two compact sessions with MSWDO Tagudin. Session 1 focuses on iteration feedback; Session 2 on SUS, training, and pilot preparation.

**References:** Manuscript Ch1–2, RAD model, [01-PROJECT-BRIEF.md](../architecture/01-PROJECT-BRIEF.md), [PHASES-OVERVIEW.md](../plans/PHASES-OVERVIEW.md)

---

## Session Overview

| Session | Focus | RAD Phase | Duration |
|---------|-------|-----------|----------|
| **Session 1** | Feedback to improve/refactor the system | System Design + Construction iteration | 2–2.5 h |
| **Session 2** | SUS, training, pilot preparation | Cutover | 1.5–2 h |

# Session 1 — Iteration Feedback

**Goal:** Gather input to improve and refactor the system before Cutover.

## Pre-Visit (1 week before)

1. Confirm attendees — 1 admin, 2–3 staff, 1–2 IT if possible
2. Send agenda (emphasize: “Your feedback will shape improvements”)
3. Send consent form if not yet signed
4. Prepare demo: laptop, local Wi‑Fi, seeded program, Display board URL
5. Print [session-1-feedback-capture.md](session-1-feedback-capture.md) or bring digital form

## Agenda

| Block | Duration | Purpose |
|-------|----------|---------|
| 1. Opening | 5 min | Intro, set expectation: feedback shapes improvements |
| 2. Requirements validation | 10–15 min | Use [requirements-validation-checklist.md](requirements-validation-checklist.md) |
| 3. Demo walkthrough | 30–35 min | Admin → Triage → Station → Display → Reports |
| 4. Hands-on trial | 35–40 min | Staff try Triage + Station; Admin tries dashboard |
| 5. Structured feedback | 20–25 min | Guided discussion; use feedback form |
| 6. Closing + next session | 5–10 min | Confirm Session 2 date |

## Prompts for Eliciting Feedback

- “If you could change one thing about [Triage / Station / Display], what would it be?”
- “Where did you expect to see [X] but couldn’t find it?”
- “Would [different label / button placement / step order] make this easier?”
- “Does this flow match how you actually work on distribution days?”
- “What would make you trust this system more for COA/reporting?”

## Note-Taking Focus

- Workflow mismatches
- Terminology (labels vs. their language)
- Mobile use (touch targets, font size, glare)
- Admin tasks (configuration, reports)
- Display/Informant readability

## After Session 1

1. Transcribe and code feedback into themes
2. Prioritize (high-impact, low-effort first)
3. Implement refactors
4. Document “Changes made based on Session 1” for Session 2
5. Schedule Session 2

## Session 1 Checklist — Before Leaving

- [ ] Feedback form completed
- [ ] Requirements validation notes
- [ ] Session 2 date confirmed
- [ ] Consent/recording handled
- [ ] Next contact confirmed

# Session 2 — SUS, Training, Pilot

**Goal:** Administer SUS, conduct training, finalize pilot. System reflects Session 1 improvements.

## Pre-Visit (1 week before)

1. Confirm attendees — ideally all 13 respondents
2. Implement Session 1 feedback; prepare “What we changed” summary
3. Prepare SUS (printed or online)
4. Prepare training materials (quick reference, optional video)

## Agenda

| Block | Duration | Purpose |
|-------|----------|---------|
| 1. Opening | 5 min | Thanks, show “What we changed” |
| 2. Walkthrough of changes | 15–20 min | Highlight 2–3 key improvements |
| 3. SUS administration | 20–25 min | Target 13 respondents |
| 4. Training | 25–30 min | Triage, Station, Admin |
| 5. Pilot coordination | 15–20 min | Program, date, roles, token prep |
| 6. Closing | 5 min | Next steps, hand off materials |

## SUS Administration

- “Standard usability questionnaire; no right or wrong answers”
- Read items aloud if needed
- For absent respondents: leave printed SUS + envelope or arrange pickup

## Pilot Coordination Outcomes

- [ ] Pilot program selected
- [ ] Pilot date confirmed
- [ ] Roles assigned
- [ ] Token preparation plan
- [ ] Server/Wi‑Fi plan for pilot day

## Session 2 Checklist — Before Leaving

- [ ] SUS completed; plan for absent respondents
- [ ] Training delivered; materials left
- [ ] Pilot date, program, roles documented
- [ ] Token prep assigned
- [ ] Follow-up and pilot-day support agreed
